## CRITICAL

### CRIT-1 — v0.6 runtime state is NOT round-tripped through save/load
**Files:** `src/persistence/serialize-game.ts` (entire file)
**Symptom:** A save/load mid-match drops EVERY one of v0.6's new GameState fields. The serializeGame body never reads — and deserializeGame never restores — `game.diplomacy`, `game.diplomacyProposals`, `game.tradeCooldowns`, `game.mythEvents`, `game.victoryRecord`, `game.portalStoneCooldowns`. The post-load game.factions registry is restored (M_V6.CARRY.SAVE-N-PLAYER), but every diplomatic relation, every pending non-aggression pact, the active myth event, the recorded 4X victory, and every portal-stone cooldown vanishes.
**Impact:** A 4X-mode match with brokered alliances + an active eclipse + a pending tribute demand → save → load = empty diplomacy state, neutral every pair, no active event. Player would see zero indication of state loss; relations they negotiated 30 minutes ago are gone.
**Fix:** Extend `GameSnapshot` interface + `SaveSnapshotSchema` Zod + `serializeGame` writer + `deserializeGame` restorer for all six fields. Use the same defensive-restore pattern that `restoreVolcanoTileMap` established (Zod-validate then assign into the already-startGame'd defaults).
**Absorbed by:** A NEW v0.7 work-unit — **M_V7.CARRY.SAVE-V6-STATE** (or fold into the in-flight M_V7.ECONOMY.REGISTRY). MUST land before M_V7.DIPLO.UI ships any save-able diplomacy interaction.

### CRIT-2 — `trainUnit` signature only allows 4 of 9 PLAYER_UNIT_TYPES
**File:** `src/game/commands.ts:329` — `role: 'Peon' | 'Footman' | 'Scout' | 'Hero'`
**Symptom:** M_PIVOT.N-PLAYER.SHARED-KIT (v0.5) added `PLAYER_UNIT_TYPES = [Peon, Footman, Trebuchet, Wizard, Healer, Ferryman, Scout, Settler, Hero]` (9 roles) and documented "every faction can train every player-kit unit." But `trainUnit` still types `role` as the 4-role v0.4 literal union. **Trebuchet, Wizard, Healer, Ferryman, Settler can never be trained via this command** — they only spawn via other paths (foundBase, etc).
**Impact:** Spec drift — the SHARED-KIT acceptance criterion ("every faction can train every player-kit unit/building") is technically NOT met. Real gameplay impact: the player can't recruit a Healer / Wizard mid-match through the build menu; AI evaluators can't train them either.
**Fix:** Widen `role: PlayerUnitType` (or `Exclude<UnitType, BarbarianUnitType>`) + add costs for the 5 missing roles to UNIT_COSTS / SUPPLY_COST tables. Re-check `canTrainComplete` covers all 9.
**Absorbed by:** NEW work-unit **M_V7.TRAIN.WIDEN-ROLES** OR fold into M_V7.ECONOMY.REGISTRY since both touch the same hot path.

## HIGH

### HIGH-1 — `tickTributeCession` only iterates legacy 2-faction `FACTIONS` const
**File:** `src/game/economy-tick-phases.ts:285`
**Code:** `tickTributeCession(game.diplomacy, FACTIONS, (f) => game.economy[f as Faction], 1);`
**Symptom:** `FACTIONS` is `['player', 'enemy']` (compile-time const). A 4-player game with `player-3` paying tribute to `player-4` never gets the cession fired — the pair-iteration in `tickTributeCession` is `O(|factionIds|^2 / 2)`, and `factionIds = ['player', 'enemy']` means `player-3|player-4` never appears in the loop.
**Impact:** Tribute mechanic completely broken for N-player matches. The substrate works (the diplomacy state holds the tributary relation), but the per-tick drain only fires for legacy slots.
**Fix:** Pass `factionIds(game.factions).filter(f => game.factions.find(...)?.kind !== 'barbarian')` so all PLAYER factions iterate. Requires GameEconomy lookup to handle N slots — directly couples to M_V7.ECONOMY.REGISTRY.
**Absorbed by:** **M_V7.ECONOMY.REGISTRY** (directive's existing work-unit).

### HIGH-2 — Camp clear reward gated to legacy 2-faction slots
**File:** `src/game/economy-tick-phases.ts:323-329`
**Code:** `if (cleared.clearedBy === 'player' || cleared.clearedBy === 'enemy') { ... }`
**Symptom:** When a 4-player match's `player-3` faction clears a camp, the deathSystem produces `clearedBy: 'player-3'`, but the reward credit `if` literal-matches only `'player'` or `'enemy'`. The wood/stone/Discovery grants never fire for player-3..N.
**Impact:** Same shape as HIGH-1 — substrate works (camp destroyed, RUINS tile, event emitted), reward credit silently no-ops for N-player slots.
**Fix:** Use `economyFor(game, cleared.clearedBy)` from the M_V7.ECONOMY.REGISTRY helper.
**Absorbed by:** **M_V7.ECONOMY.REGISTRY**.

### HIGH-3 — `wonderTimers: Record<Faction, number>` blocks N-player wonder victory
**File:** `src/game/game-state.ts:335,954` + `src/game/economy-tick-phases.ts:377-378`
**Symptom:** `wonderTimers.player` / `wonderTimers.enemy` are the only two slots tracked. A 4X-mode 6-player game where `player-4` builds a Wonder cannot win via the wonder-completion path because the timer never decrements for player-4 (no key).
**Impact:** Wonder-based victory limited to legacy slots. Substrate broken for the user's headline 4X feature.
**Fix:** Lift `wonderTimers` to `Record<FactionId, number>` (Map<FactionId, number> avoids the dense allocation). Initialize from `factionIds(game.factions)` at startGame.
**Absorbed by:** **M_V7.ECONOMY.REGISTRY** (same lift pattern as economy).

### HIGH-4 — FactionConfigSchema permits duplicate ids
**File:** `src/persistence/serialize-game.ts:432-440` (FactionConfigSchema)
**Symptom:** The schema validates each row but doesn't enforce id uniqueness across the array. A tampered save with `factions: [{id:'player', ...}, {id:'player', ...}]` loads cleanly; `findFaction()` returns the FIRST match, silently shadowing the second.
**Impact:** Save tamper attack — two `player` slots could create undefined N-player behaviour (zone state, economy, AI all keyed by the first id; second's entity carries the same id and would inherit the other's resources at runtime).
**Fix:** Add `.refine(arr => new Set(arr.map(f => f.id)).size === arr.length, 'duplicate faction ids')` to the array schema.
**Absorbed by:** **M_V7.CARRY.SAVE-V6-STATE** (the CRIT-1 follow-up).

## MEDIUM

### MED-1 — `factionOverride` lies to TypeScript via `as 'player' | 'enemy'` cast
**Files:**
- `src/entities/character-factory.ts:118` — `FactionTrait({ faction: faction as 'player' | 'enemy' })`
- `src/world/barbarian-camps.ts:194` — same shape
**Symptom:** Both casts are deliberate — koota traits accept any string at runtime even though the trait type narrows to the literal union. The CAST WORKS but future maintainers reading the trait sig + writing `if (faction === 'player')` switches will silently miss `barbarian-camp-1`. Architectural debt — the runtime / type-system divergence is now load-bearing across 2 files.
**Impact:** No runtime bug today. Trap waiting for a future commit.
**Fix:** Either (a) widen `FactionTrait.faction` to `string` and accept the `Record<Faction, X>` cascade, OR (b) introduce a `BrandedFactionId` type with a `runtimeFactionId(s: string): BrandedFactionId` constructor that the casts use instead, making the divergence explicit and grep-able.
**Absorbed by:** Track as v0.8 follow-up — the lift is invasive and v0.7 has higher-priority correctness work.

### MED-2 — `victory-conditions.ts` `countDiplomaticTies` parses relation keys naively
**File:** `src/game/victory-conditions.ts:63` — `const [a, b] = key.split('|');`
**Symptom:** The diplomacy `relationKey(a,b)` joins ids with `|`. Today's ids (`player`, `enemy`, `player-3`, `barbarian-camp-1`) don't contain `|`. But there's no schema guard forbidding `|` in a future factionId (e.g. user-supplied alphabet in a friendly multiplayer match would happily include `|`). `split('|')` would return 3+ pieces and silently mis-parse.
**Impact:** Defensive — current ids are safe.
**Fix:** Either (a) refactor RelationEntry to carry the pair ids directly (eliminate the parse), or (b) Zod-validate factionId schema with `.regex(/^[a-z0-9-]+$/i)` rejecting `|`. (a) is the right answer because it also makes `countDiplomaticTies` O(1) per entry instead of O(N) key.split.
**Absorbed by:** Fold into M_V7.ECONOMY.REGISTRY scope (same architectural pass).

### MED-3 — No save-snapshot version bump for v0.6 fields
**File:** `src/persistence/serialize-game.ts:83` — `SNAPSHOT_VERSION = 2`
**Symptom:** v0.6 didn't bump SNAPSHOT_VERSION to 3 even though the GameState shape grew 6 fields. A future v0.7 save that adds diplomacy will need version 3 (the schema bump establishes "v3 has diplomacy; v2 doesn't"). Today's v2 saves and v3 saves are mutually incompatible at the SCHEMA level, but the version stayed at 2.
**Impact:** A v0.5 save (also version=2) can't be distinguished from a v0.6 save (also version=2) by the migration framework. If v0.7 saves are version=3 and v0.6 / v0.5 are both version=2, the version=2 migration arm has to handle BOTH shapes.
**Fix:** Bump SNAPSHOT_VERSION to 3 in the CRIT-1 fix commit; add a version=2 → version=3 migration that initializes empty diplomacy / mythEvents / cooldowns from `createDiplomacyState()` etc.
**Absorbed by:** **M_V7.CARRY.SAVE-V6-STATE** (same commit as CRIT-1).

### MED-4 — RUINS / PORTAL_STONE biomes have no Decoration palette entry
**File:** `src/world/Decoration.tsx:44` — `PALETTES: Partial<Record<BiomeType, BiomePalette>>`
**Symptom:** The `Partial<>` shape gracefully skips RUINS / PORTAL_STONE (palette is undefined → `if (!palette) continue;`). Visually those tiles render as just a colored hex with no props — which IS the spec intent for PORTAL_STONE (a swirl tile, no overlay) but is THIN for RUINS (the spec said "old camp remains" — suggests a couple of ruined-wall fragments).
**Impact:** Visual polish gap for RUINS — feels under-developed.
**Fix:** Add a RUINS palette entry to Decoration.tsx PALETTES with a low-density `structures.gate-stone` + `structures.fountain` fragment scatter. PORTAL_STONE deliberately has no decoration (the swirl + violet color carry the surface).
**Absorbed by:** **M_V7.VISUAL.BATTERY** (visual polish + baseline lock).

### MED-5 — MYTH event dispatchers exist for only 1 of 5 events
**File:** `src/game/myth-events.ts` (only `applyHarvestFestival` shipped)
**Symptom:** `fireMythEvent` records `state.active` but the runtime calls no per-event dispatcher. `applyMeteorStrike`, `applySolarEclipse`, `applyWildlifeMigration`, `applyOracleVision` don't exist. The events FIRE (state mutates) but produce no in-game effect except the harvest-festival case (which itself isn't auto-dispatched on fire — the caller has to invoke `applyHarvestFestival` separately).
**Impact:** 4 of 5 MYTH events are pure-config no-ops in production. The user sees "Solar Eclipse!" in the (eventual) HUD but no actual eclipse happens.
**Fix:** Add the 4 dispatchers + wire `fireMythEvent` to invoke the matching dispatcher.
**Absorbed by:** **M_V7.MYTH.EFFECTS** (directive's existing work-unit).

## Architectural recommendations

1. **The literal-union `Faction` type has become structurally load-bearing in the wrong way.** Two cast sites (character-factory + barbarian-camps) lie to TypeScript; tributary cession + camp reward + wonder timers all silently miss N-player slots. **The right fix is M_V7.ECONOMY.REGISTRY** — widen the storage, not the trait. Keep `type Faction = 'player' | 'enemy'` for the FactionTrait literal (covers the 2 narrative slots), but every `Record<Faction, X>` lifts to `Map<FactionId, X>` or `Record<string, X>` with a helper accessor.
2. **`game.diplomacy.relations: Map<key, RelationEntry>` is using the wrong storage shape.** The key encodes the pair; queries parse the key back. **The right shape is `RelationEntry & { a: FactionId; b: FactionId }`** so consumers iterate `for (const entry of game.diplomacy.relations.values()) { ... }` without parsing. Fold into M_V7.ECONOMY.REGISTRY pass.
3. **Save-snapshot versioning** — v0.6's six new fields require a SNAPSHOT_VERSION bump to 3 + a v2→v3 migration arm. Today version=2 covers two structurally different shapes. Fold into M_V7.CARRY.SAVE-V6-STATE.
